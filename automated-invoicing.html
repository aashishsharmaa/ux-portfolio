Case Study 02 / Amazon, Payee Central, 2017-2018

Automated invoicing for high volume payees

A self-service interface for high volume payees, so that they can receive their payments faster, at less cost and effort.

Context

Payees invoice Amazon to receive payments.

Payees are suppliers who sell goods & services to Amazon, and then send invoices to receive payments. These invoices, mostly in PDF format, were sent to Amazon via email. Amazon used to process these invoices through legacy systems.

Amazon created Payee Central, using which payees could create invoices via webform.

Webform involved manually filling in invoice details, requiring time, effort and cost, and thus was suitable for payees with low invoice volume.

Automation was needed to process invoices for high volume payees.

A small number of payees (~2%) did business at high volume, nearly ~70% of the total invoice processed. Existing methods of invoice processing resulted in higher time, effort and cost for them. Automation was needed to process invoices for high volume payees.

EDI - Electronic Data Interchange - is an industry standard to automate invoice processing. It requires a one-time setup, after which invoice files can be directly transferred between servers of two companies.

Context

Business and User Goal.

Business goal: Onboard 15 payees.

The business goal of this project was to onboard 15 payees with EDI capabilities to Payee Central.

UX goal: Create self-serivce interface for EDI setup.

The UX goal was to create a self-service interface using which payees can create their EDI setup, and do maintenance activities when required, so that they are invoice-ready at all times.

Primary user

Software developers on Payee teams were the primary users.

Their goal was to create and maintain an EDI setup which was invoice-ready at all times.

To create an EDI Setup, they would create EDI connections and test EDI files. For maintenance, they would update connection certificates, and verify more EDI files.

The users were well trained at their job, but required some data from Amazon to complete their work.

Primary users were software developers.

Understanding backend

What's an invoice-ready EDI Setup?

The EDI setup was invoice ready, when there was -

  • at least 1 active EDI connection, and
  • at least 1 verified EDI test file

Through discussion with team, did a deep-dive on -

  • how data would flow for a successful setup
  • various error scenarios that were to be handled

Invoice-ready EDI setup.

Plan

Given the complexity, the project was divided into multiple stages.

The plan was to first build the basic back-end infrastructure to get the business started with a single EDI connection and file type, and later scale to support multiple EDI connections and file types.

Multi-stage plan.

Solution - Stage 01

Communicating error messages

In stage 1, there was no interface planned.

Amazon developers were to receive EDI connection details & test files from Payee developers via email. Then, manually create an EDI connection, and add the test file to the system being built for verification.

The back-end system would send an email to Payee developers notifying success or failure.

Since this was external communication to end users, my contribution was in writing the copy for the templates.

Wrote copy for email templates.

Soon, we realised the number of errors was far more than initially expected, and an email was not an ideal solution.

Suggested creating a separate page to report the errors, and sending the page link via email.

Created an error page for resolution.

Solution - Stage 02

Testing EDI Files

In stage 2, testing an EDI file was to be done via the interface, while using the EDI connection created in stage 1.

The initial expectation was a simple process with 2 actions:

  • Provide a sample EDI file to payee developers.
  • Ask them to upload their EDI test file.

However, we built a complicated flow.

To increase the success rate of test files, developers suggested leveraging the previously paid invoices, as these payees were already doing business with Amazon via other services.

Thus, the simple flow became complex.

Expected a simple flow.

Built a complicated flow to leverage previously paid out invoices.

To handle the complexity -

  • Used progressive disclosure to show only the relevant input fields to the user
  • Remotely tested low fidelity wireframes with users, to see whether they were able to understand the interaction.

Used progressive disclosure, and remotely tested wireframes with users.

Once we were satisfied that the users followed the interaction, the next step was to create a flow to repeatedly test EDI files.

Designed a flow to repeatedly test files.

Users could test new files, and view status of previous tests.

For handoff -

  • Detailed the wireframes in Amazon's UI style.
  • Illustrated the flow with success and error messaging.

For dev-handoff, illustrated the flow with visual styles and messaging.

Solution - Stage 04

Multiple EDI Connections & File Types

In stage 4, we needed to support multiple connection and file types to expand operations in multiple geographies.

It was the most challenging stage in this project, and we faced 2 challenges.

First, our understanding of adding multiple file types was wrong.

Use of check-boxes for multi-selection was fine for creating multiple connections, but for testing file types we found that radio buttons were needed to single select a file type.

And second, we had a new definition of user goal.

To ensure a high rate of test file verification, developers on our team suggested requiring at least 3 verified files.

To test multiple file types, we thought of 2 approaches-

  • Reuse existing UI, but users would have to test one file at a time.
  • Try a new approach, where multiple files could be uploaded in one go.

Reuse Existing UI - Users had to test files one at a time.

New UI - Users could upload multiple test file in one go.

These looked simple during whiteboarding. But complexity emerged when detailed in medium fidelity.

We had to -

  • Rely on a lot of messaging to explain what a user could do next, or
  • Handle complicated drop-downs.

Lot of messaging was needed for explanation.

Complicated dropdowns were difficult to understand.

We knew we were wrong, and users also dismissed these solutions.

Users could test only one file at a time.

We had ignored how they would test the files in real life.

Testing even one file took repeated efforts, and they could test only one file at a time.

Used radio buttons for testing a single file type.

So, we replaced the check-boxes with radio buttons on upload test file screens, while keeping the check-boxes for creating EDI connections.

Focused on user goal of successful test files.

To solve for at least 3 files, the solution lied in focusing on the user goal, and not their tasks.

So, instead of writing "Progress bar" for the number of files being uploaded, replaced it with "Successful test files" for number of verified files.

Tried various possibilities, and finally reused a progress bar to show the number of successful test files.

Lastly, we were to show Amazon's EDI specifications. I tried some visual variations, and kept the hierarchy of various elements in mind to arrive at the final solution.

Variation #1

  • Visually scattered
  • Information not in the order of user need
  • Too many info icons competed for attention

Variation #2

  • Visually less scattered
  • Information not in the order of user need
  • Important data inside a banner

Variation #3

  • Visually balanced on page
  • Information in the order of user need
  • Emphasis on important data

The payees could now invoice in different regions using multiple connection types & file types.

Payees could invoice in different regions, using multiple connection types & file types.

Solution - Stage 05

Dashboard to see invoice-readiness

A dashboard was needed by the Payee Account Managers to -

  • See the countries where they can invoice, according to a file type.
  • Fix maintenance issues by updating connection certificates, or uploading new test files.

Created a few visual variations, tested with users, and iterated until there was a right fit.

Users could see where they can invoice.

Putting everything together.

With all these sections on the set page, we ended up with a long page.

Launch & Learnings

For the launch, made an infographic informing the stakeholders about the project's impact.

In terms of UX processes, we learned -

  • Importance of testing with wireframes before putting in development effort.
  • To focus on user goals, and not get lost into details of user tasks.

In term of Amazon's Leadership Principles, the learnings were -

  • Deep dive on complexity.
  • Simplify, simplify and simplify.
